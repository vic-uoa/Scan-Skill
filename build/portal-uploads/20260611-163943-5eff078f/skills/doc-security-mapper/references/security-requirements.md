# 安全需求

共 64 条安全需求，结构：序号｜需求名称｜需求设计

## 数据采集与存储安全

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 1 | 应用系统应遵循最小采集原则，避免超范围采集 | 应用系统采集敏感数据时应遵循最小采集原则，确保没有超范围采集额外的数据。 |
| 2 | APP、WEB、小程序等客户端本地不应存放敏感数据 | 依据最小够用原则存储数据，不在 APP、WEB、小程序本地存放敏感信息。 |
| 6 | 应避免客户端缓存敏感信息，确保数据加密保护 | 避免客户端缓存含敏感信息的网页。需对数据进行加密保护，设置 `Cache-Control: no-store` 和 HTTP 头 `Pragma: no-cache`。 |
| 51 | 小程序应不将敏感数据存放在代码或配置文件中 | 依据最小够用原则存储数据，敏感数据不应以明文、注释、可逆编码等形式出现在小程序文件内。 |
| 52 | 敏感数据传输加密与安全存储 | 敏感数据传输应优先使用商用密码技术，使用安全传输协议（HTTPS/SFTP/SSL VPN），禁止 Cookie 明文存储敏感数据。 |
| 54 | 身份鉴别数据存储 | 传输身份鉴别数据应加密存储并禁用弱加密算法。 |
| 55 | 敏感数据跨网访问应脱敏处理 | 敏感数据原则上只能在业务网访问，如需在低安全等级环境访问，应对个人敏感信息进行脱敏。 |
| 56 | 身份鉴别数据应限制导出及加密展示 | 传统身份鉴别数据原则上不应导出，不允许明文展示客户的身份鉴别数据。 |
| 61 | 使用对象存储服务应满足权限最小化原则并加密脱敏 | 若使用对象存储服务，需满足数据访问控制最小化原则和数据分类存储要求。 |
| 62 | 采集敏感数据应避免本地存放并保障真实性合法性 | 通过 App/Web 等客户端采集敏感数据时，业务处理结束后应删除本地存放的敏感数据。 |
| 63 | 应用系统采集个人敏感数据应获取用户授权 | 应用系统向个人采集敏感数据前，应确保个人已经知情同意。 |
| 64 | 身份鉴别数据采集应动态认证保障真实性完整性 | 采集传统身份鉴别数据时，不应通过人工方式采集，建议对采集全过程进行持续动态认证。 |

## 日志与审计安全

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 3 | 对关键操作和敏感操作进行详细日志记录并定期审查 | 对关键操作（如登录、密码修改等）进行详细日志记录，并定期审查分析以便及时发现潜在安全风险。 |
| 4 | 日志不应保存敏感信息 | 日志不应保存敏感信息（例如：会话标识、密码和客户信息、私钥信息等）。 |
| 5 | 敏感数据操作时，应记录全面的审计日志信息 | 对于用户端、敏感数据处理、后台权限增删改操作应记录敏感数据操作审计日志，日志内容至少应包括操作用户、操作时间、登录 IP、操作类型、操作内容、操作结果等。 |
| 39 | 记录登录日志应防范认证过程不可追溯、审计 | 应记录用户身份认证日志，日志内容至少应包括：认证用户、认证日期和时间、用户 IP、认证是否成功。 |
| 57 | 日志记录需脱敏操作并避免存储密码 | 日志中不应记录密码信息，应记录数据采集、访问、下载、删除、打印等类型的敏感数据操作日志。 |

## 通信传输安全

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 7 | 数据出境需实施安全评估或网信办申报方可投产 | 涉及数据出境，应依据相关管理规定实施安全评估；涉及个人信息、重要数据出境需向网信办申报。 |
| 9 | 互联网开放 API 应使用 TLS/HTTPS 等安全传输加密 | 面向互联网开放 API 时，应使用 TLS 等安全通道加密传输，建议使用 TLS 1.2 及以上版本。 |
| 60 | 跨网数据传输应控制范围并评审 | 系统数据跨网段传输必须明确区分访问范围，遵循业务最小必要原则，数据内容变动需重新安全评审。 |

## 文件安全

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 10 | 客户端应保护认证信息并禁用不必要的配置及组件以增强安全 | 检查上传文件类型和大小，上传文件存放在 Web 应用范围之外，上传路径需过滤特殊字符。 |
| 11 | 文件上传防范任意文件上传造成的各类攻击 | 检查上传文件类型和大小，判断是否为可执行文件，上传文件存储在 Web 应用范围之外，禁止任何文件执行。 |
| 12 | 文件上传防范应用目录信息泄露 | Web 应用各目录均未开启目录浏览功能，不存在通过其他信息扫描泄露目录名称的问题。 |
| 13 | 文件下载防范应用目录信息泄露 | Web 应用各目录均未开启目录浏览功能，不存在通过其他信息暴露目录名称的问题。 |
| 14 | 文件下载防范应用目录遍历 | 应避免直接通过文件名提供文件下载功能，建议采用随机 ID 方式，对包含 `../` 等特殊字符的路径进行拒绝。 |

## 访问控制与权限管理

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 8 | 错误页面应定制并禁用默认以防信息泄露 | 必须使用定制的错误页面返回给客户端，避免暴露容器版本信息和操作系统信息。 |
| 15 | 角色权限管理应防范水平越权和其他越权行为 | 账号登录的用户角色验证和授权机制必须在服务器端有效，确保只有经过授权的用户才能访问相应功能。 |
| 58 | API 输入应该严格校验参数及其来源的合法性 | 应严格检查输入数据的格式、类型、长度、范围等要素，对非预期的数据进行拒绝，使用白名单或黑名单方式验证。 |
| 59 | API 需要有访问控制策略，包括认证、会话管理等 | 实施接口访问控制策略，对敏感数据和功能的访问实施控制，对需授权访问的 API 严格认证调用方身份。 |

## 用户认证安全

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 16 | 用户注册验证防止恶意注册 | 用户注册验证应防止密码验证码/图形验证码被绕过，避免恶意用户大量注册账户。实名注册时至少通过姓名与身份证号码匹配度验证。 |
| 17 | 用户注册验证防止窃取用户信息或密码 | 同一手机号或用户名重复注册多个账户时，应避免恶意用户通过多次注册已存在账户来获取信息或重写密码。 |
| 18 | 首次密码设置防止弱口令 | 使用复杂密码，包含大写字母、小写字母、数字、特殊字符等组合，密码长度不少于 8 位。 |
| 19 | 用户注册验证防止用户信息泄露 | 用户首次登录系统时，系统提供默认密码场景，应强制用户修改默认密码，且默认密码应具有复杂度。 |
| 20 | 匿名登录防止信息泄露或无法追踪关键操作 | 移动端平台账号为绑定登录，手动绑定后至少设置连续两次的 4 个点。 |
| 21 | 密码错误防止暴力破解密码 | 密码的强度应符合国家密码管理局的要求。 |
| 22 | 密码错误防止账户枚举破解密码 | 密码的强度应符合国家密码管理局的要求。 |
| 23 | 忘记密码/重置密码防止用户信息泄露 | 密码的强度应符合国家密码管理局的要求。 |
| 24 | 忘记密码/重置密码防止密码重置逻辑绕过 | 密码重置前应通过短信验证码、用户注册信息校核等方式确认用户身份。新设置密码应使用复杂密码，禁止使用弱密码，业务流程禁止特定参数直接跳转。 |
| 25 | 用户登录失败防止会话劫持 | 用户未活动超过 30 分钟、成功更改密码或权限变更后，应使原有会话失效；登录成功后应自动更新会话标识。 |
| 26 | 用户登录失败多点登录需有效控制 | 移动金融客户端应设置合理的多点登录策略，如提示登录信息或登出先登录的账户等。 |
| 27 | 用户登录密码展示防止用户信息泄露 | 密码的默认展示应使用特殊字符代替明文。 |
| 28 | 用户登录动态口令需防止相关风险 | 动态口令长度不应少于 6 位，需进行输入错误次数限制并锁定，应设置动态口令有效期。 |
| 29 | 修改密码密码展示应防范用户信息泄露 | 密码的默认展示应使用特殊字符代替明文。 |
| 30 | 修改密码响应防范相关安全风险 | 修改密码前应对用户身份进行重新验证，应正确检查原始密码，新密码不应与原密码相同，应使用复杂密码。 |
| 31 | 修改密码响应防范预留手机号被篡改 | 客户通过网上银行变更预留手机号，应采用双因素身份认证验证当前用户身份，并通过反向拨打手机号或短信验证码确认。 |
| 32 | 人脸识别应防范被绕过的风险 | 采用的人脸识别算法需具备活体检测能力。面向互联网的人脸识别不能作为单独认证方式，需对人脸信息加密传输。 |
| 33 | 短信验证码应防范被破解或冒用 | 短信验证码 10 分钟内使用则失效，长度不应少于 6 位，认证通过后验证码应失效。 |
| 34 | 短信验证码应防范被暴力破解 | 发送验证码时，应同业务处理环节相关联，并对同一手机号码、IP 地址在同一时间内获取验证码进行限制。 |
| 35 | 图形/滑动验证码应防范仿冒、泄露或自动识别 | 图形验证码应在一定连续错误次数后出现并具备抗自动识别能力，防止被自动化工具绕过。 |
| 36 | 采购人脸识别应防范仿冒 | 面向互联网的人脸识别不能作为单独认证方式，人脸识别算法需具备活体检测能力。 |
| 37 | 退出登录应防范会话劫持 | 退出登录时，应向服务器发起会话结束请求，使当前会话状态失效。 |
| 38 | 退出登录应防范会话劫持 | 退出登录功能应主动清除服务端会话并使会话标识失效，防止退出后继续访问。 |

## 支付安全

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 40 | 特殊支付应防范恶意篡改 | 用户确认支付后才可执行支付功能，除转账或支付金额、币种执行检查，防止金额篡改。 |
| 41 | 特殊支付应防范高频攻击 | 为每个支付请求分配唯一交易号，限制支付频率，使用双因素认证、使用令牌控制防止重放攻击。 |
| 42 | 二维码支付应防止密码被猜解 | 应对验证的支付密码进行保护，不应与登录密码相同。 |
| 43 | 二维码支付应防止支付中间人攻击 | 确保二维码的使用环境可信，控制二维码有效期，限制异常访问和高频访问。 |
| 44 | 二维码支付防范中间人攻击 | 应在客户端中判断账户识别信息完整性，不可直接通过链接跳转进行登录。 |
| 45 | 二维码支付防范客户端敏感信息泄露 | 账户信息不应在扫码设备和系统中存储，互联网环境下二维码不应明文显示传输。 |

## 业务逻辑安全

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 46 | 业务类功能防止信息篡改 | 如抽奖场景有次数限制，应限制业务调用量并在服务端验证同一用户合法性的抽奖约束。 |
| 47 | 业务类功能防止信息篡改 | 应分别对业务操作进行授权检查，限制认证对象和授权范围。 |
| 48 | 交易订单类功能防止信息篡改 | 提交订单时应防范业务逻辑篡改，模拟订单、退货、金额变更等场景应根据字段控制用户权限。 |

## 移动端安全

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 49 | 移动客户端应检测运行环境是否安全 | 客户端在打开和运行时，检查运行环境安全性，确认是否存在非常规攻击工具，确认是否在模拟器中运行。 |
| 50 | 移动客户端应检测自身代码完整性防范后门工程 | 通过哈希等校验代码完整性，采用混淆、加密、脱壳等防止被攻击和后门。 |

## 密钥管理

| 序号 | 需求名称 | 需求设计 |
| --- | --- | --- |
| 53 | 密钥管理应加强保护 | 若使用浏览器插件类密钥，需接口对浏览器插件进行最小权限控制，服务端使用硬件加密模块或密码服务平台存储密钥分量。 |
