# 整改建议库

本文件用于把规则建议从“泛泛提示”升级为可执行整改动作。报告中的建议应优先按照 `rule_id + file_role + decision` 组合生成，避免所有场景都输出同一段模板。

## 建议生成原则

| 维度 | 处理方式 |
|:---|:---|
| `confirmed` | 给出明确修复动作、验证方式和阻断原因 |
| `probable` | 给出核实点和最小整改方向 |
| `needs_review` | 给出人工复核要看的证据清单 |
| `low_risk_notice` | 给出治理优化建议，不写成漏洞结论 |
| `false_positive` | 建议沉淀误报类型，不进入用户最终报告 |

## 高频风险建议模板

### 敏感文件与凭据

- 最小修复：移除对宿主机 `.env`、`.ssh`、`.pem`、`.key`、云凭据文件的默认读取。
- 推荐做法：由用户显式传入临时测试凭据路径，并校验路径位于当前 workspace 或受控测试目录。
- 验证方式：扫描报告中不再出现真实凭据路径；运行时不会读取用户 HOME、浏览器 profile 或云凭据目录。

示例：

```text
错误：默认读取 ~/.ssh/id_rsa
建议：要求用户传入 ./test-fixtures/temporary-key.pem，并校验该路径属于当前项目目录
```

### Token、Cookie、认证上下文

- 最小修复：删除报告、日志、fixture、示例中的真实 token、cookie、session、用户号。
- 推荐做法：保留字段类型，值统一写成 `<token>`、`<masked>`、`user-***`。
- 验证方式：报告中仍能说明风险，但不可复现真实请求。

示例：

```text
错误：Authorization: Bearer eyJ...
建议：Authorization: Bearer <masked-token>
```

### 报告泄露专项

- 最小修复：隐藏真实域名、接口路径中的业务对象号、事件号、taskId、request_body、cookie、token。
- 推荐做法：保留接口分类、HTTP 方法、认证方式和风险结论。
- 验证方式：报告可用于审计，但不能直接调用内部接口或复现用户请求。

### 安全测试 Payload

- 最小修复：把 payload 放入 `security-tests/`、`payloads/` 或 `examples/`，并标明授权范围。
- 推荐做法：补充测试环境、测试账号、速率限制、禁止生产环境的说明。
- 验证方式：扫描器将其降级为复核项或低风险提醒，而不是高危漏洞。

### 命令执行与动态代码

- 最小修复：禁止拼接用户输入执行 shell，删除 `eval/exec/new Function`。
- 推荐做法：使用白名单命令表、参数化 API、固定临时目录。
- 验证方式：命令参数不可被外部输入控制；清理动作限定在当前 workspace。

### 数据库与账务变更

- 最小修复：写操作必须有 where、事务、影响行数确认和回滚。
- 推荐做法：默认连接隔离测试库，禁止内置生产连接。
- 验证方式：扫描报告不再出现生产 JDBC、无条件 delete/update、账务字段批量修改。

### Skill 治理

- 最小修复：补齐 owner、version、changelog、输入输出、环境边界、验证方式、清理回滚。
- 推荐做法：把个人脚本治理成团队资产，说明默认不访问生产、不读取真实会话、不写公共目录。
- 验证方式：META 类问题降为通过或低风险提醒。
